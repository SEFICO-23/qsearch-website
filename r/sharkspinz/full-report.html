Executive Summary

The Bottom Line

The platform is catastrophically vulnerable at the application layer. An unauthenticated attacker can generate unlimited platform credit, predict every game outcome, bet on any user's account, spoof any @sharkspinz.com email, and read the majority of 56 database tables — all using only the API key hardcoded in the frontend JavaScript and a standard web browser.

The platform's strongest defense is its credential management: the JWT secret, database password, SSH credentials, and Studio login all survived extensive brute-force testing including the full 14.3M-entry rockyou.txt wordlist. Row-Level Security on configuration tables also held against all 14 bypass techniques.

Important caveat: While an attacker can generate unlimited platform credit and manipulate all game outcomes, the withdrawal chain currently breaks at step 4 of 7 — no withdrawal processor exists. However, the moment a withdrawal handler is implemented, pre-staged "completed" withdrawal records in the database would be processed.

La piattaforma è catastroficamente vulnerabile a livello applicativo. Un attaccante non autenticato può generare fondi illimitati, prevedere ogni risultato di gioco, scommettere sull'account di qualsiasi utente, falsificare qualsiasi email @sharkspinz.com e leggere la maggior parte delle 56 tabelle del database — il tutto usando solo la chiave API codificata nel JavaScript frontend e un browser web standard.

La difesa più forte della piattaforma è la gestione delle credenziali: il segreto JWT, la password del database, le credenziali SSH e il login Studio hanno resistito a test di forza bruta estesi, inclusa la lista completa di 14,3 milioni di voci rockyou.txt.

Avvertenza importante: Sebbene un attaccante possa generare credito illimitato e manipolare tutti i risultati di gioco, la catena di prelievo attualmente si interrompe al passaggio 4 di 7 — non esiste un processore di prelievo. Tuttavia, nel momento in cui viene implementato un gestore di prelievi, i record "completati" pre-inseriti nel database verrebbero elaborati.

Security Findings

All 36 Findings

Critical Findings

CriticalSS-001CVSS 9.8Financial

Infinite Money — 3 Independent Paths

Three completely independent methods allow unlimited financial manipulation. An unauthenticated attacker can add unlimited funds using only the anon key hardcoded in frontend JavaScript.

Path A — Negative Balance Attack (No Auth Required)

The deduct_balance_atomic RPC function accepts negative amounts. Calling with p_amount: -100 triggers balance = balance - (-100) = balance + 100. Two calls added $200 to a zero-balance test account.

# No authentication beyond the public anon key
curl -s "https://sharkspinz.com/rest/v1/rpc/deduct_balance_atomic" \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"p_user_id": "<USER_ID>", "p_amount": -100}'
# Response: true — balance increased by $100

Path B — Direct Balance PATCH (Auth Required)

Any authenticated user can PATCH their own profile's balance field to any value. RLS restricts row access but does NOT restrict column access. Balance set from $0 to $999,999 confirmed.

curl -s "https://sharkspinz.com/rest/v1/profiles?user_id=eq.<ID>" \
  -X PATCH -H "Authorization: Bearer <TOKEN>" \
  -d '{"balance": 999999}'
# Response: [{"balance": 999999, ...}]

Path C — Wagering Requirement Bypass

total_wagered field is also PATCHable, allowing bypass of any wagering requirements before withdrawal.

Remediation

  • Add IF p_amount <= 0 THEN RAISE EXCEPTION to deduct_balance_atomic
  • Create column-restricted RLS policy on profiles preventing modification of balance, total_wagered, total_won, vip_tier, and banned
  • Move all financial operations to server-side Edge Functions with proper auth
CriticalSS-002CVSS 9.8Gambling Integrity

Server Seed Plaintext Leak — Every Game Predictable

The process_single_bet RPC function returns the server_seed in plaintext. The server seed determines every game outcome (dice, limbo, roulette, wheel, plinko). The entire "provably fair" guarantee is destroyed.

// API response includes the secret seed
{
  "success": true,
  "result": "win",
  "payout": 1.96,
  "server_seed": "a1b2c3d4e5f6...",  // PLAINTEXT SECRET
  "new_balance": 10098.00
}

Additionally, the provably_fair_seeds table is readable and exposes server seeds even when revealed: false. Combined with publicly readable game weights (SS-018), an attacker can calculate exact expected values and optimal strategies for every game.

Remediation

  • NEVER return server_seed in bet responses — only after seed rotation
  • Add RLS to provably_fair_seeds restricting reads to revealed seeds only
  • Implement proper provably fair protocol: hash(server_seed) before game, reveal only after game + seed rotation
CriticalSS-003CVSS 9.8Access Control

Anonymous Betting on Any User's Account

process_single_bet accepts an arbitrary user_id parameter and is callable with the anon key. An attacker can place bets on ANY user's account, draining their balance. Combined with predictable outcomes (SS-002), the attacker controls whether bets win or lose.

Remediation

  • Add auth check to process_single_bet — validate that the caller's JWT sub matches the user_id parameter
  • Reject anon-key calls entirely
CriticalSS-004CVSS 9.1Email Security

Email Spoofing — No DMARC/DKIM (8/8 Delivered)

sharkspinz.com has zero email authentication: SPF softfail only, no DKIM, no DMARC. 8/8 spoofed emails delivered to test mailboxes. 7 unique @sharkspinz.com personas impersonated including billing@, security@, admin@, marco@.

Three attack scenarios demonstrated:

  • User phishing — Fake withdrawal notification from billing@sharkspinz.com using real leaked email template
  • Admin phishing — Security alert from security@sharkspinz.com citing real infrastructure details
  • Multi-persona impersonation — 5 simultaneous personas, all delivered

Remediation

  • Add DMARC (p=reject), DKIM signing, and harden SPF to -all (hardfail)
  • Cost: $0, time: 30 minutes
CriticalSS-005CVSS 8.6Data Exposure

Database Tables Readable incl. Payment Methods & Bonus Codes

The Supabase anon key (hardcoded in frontend, 10-year expiry) grants SELECT access to most of the 56 database tables. Currently exposed: payment methods with IBAN/BIC/SWIFT, all bonus codes, email templates, blockchain configs, game economics, site configuration.

When the platform goes live with real users, exposed tables will include: KYC documents, wallet addresses, transaction histories, API keys, and crypto treasury private keys.

Remediation

  • Audit ALL RLS policies on every table
  • Apply SELECT restrictions: anon should access only public data (game catalog, site branding)
  • Financial tables require service_role or admin-only access
CriticalSS-007CVSS 8.6Information Disclosure

Kong Exposes Full Swagger Schema (87 Endpoints)

Kong on port 8000 returns the complete PostgREST OpenAPI specification with the anon key: 87 endpoints across 56 tables. Schema reveals fields like private_key, totp_secret, server_seed, and api_key.

Remediation

  • Firewall port 8000 — block external access entirely
  • If API gateway is needed externally, restrict the OpenAPI schema endpoint
CriticalSS-008CVSS 8.6Authentication

Auto-Confirmed Signups (Unlimited Account Creation)

Supabase auth configured with mailer_autoconfirm: true. Any signup request instantly creates a verified account — no email verification, no CAPTCHA, no rate limiting. Enables mass account creation for bonus abuse, self-referral chains, and bot armies.

Remediation

  • Disable mailer_autoconfirm — require email verification
  • Add CAPTCHA to signup flow
  • Implement rate limiting on auth endpoints
CriticalSS-009CVSS 8.6Data Integrity

Transaction Record Injection (Audit Trail Forgery)

Any authenticated user can INSERT records into the transactions table with arbitrary type, amount, and status. Fake deposit/withdrawal records destroy audit trail integrity. Combined with SS-001 and SS-014, enables complete fraud chain.

Remediation

  • Add RLS INSERT policy on transactions — restrict to service_role only
  • All transaction creation should go through server-side Edge Functions with proper validation

High Findings

HighSS-006CVSS 7.4Cryptographic

pgjwt sign()/verify() Publicly Callable — JWT Secret Oracle

PostgreSQL pgjwt extension functions exposed as public RPC endpoints. verify() returns valid true/false at ~93 req/sec with zero rate limiting. If the secret is ever recovered, sign() forges a service_role JWT — total database compromise. Current mitigation: secret survived 14.3M+ attempts.

HighSS-010CVSS 7.5Social Engineering

Notification Injection (In-Platform Phishing)

Any authenticated user can INSERT notifications targeting ANY user with type: "system", making them appear as official platform messages. Includes a link field for redirect-based phishing.

HighSS-011CVSS 8.1Infrastructure

CVE-2024-6387 regreSSHion (Unauthenticated Root RCE)

OpenSSH 9.6p1 confirmed vulnerable to the regreSSHion race condition. Practical exploitation on 64-bit systems has extremely low probability (~10,000 connections over 6-8 hours, ASLR-dependent). Recommendation: Upgrade to 9.8p1+.

HighSS-012CVSS 8.6Infrastructure

Adminer Publicly Accessible (Port 3000) incl. CVE-2026-25892 DoS

Full-featured database management tool accessible from the internet with no IP restriction, no VPN, no auth proxy. Pre-configured with internal hostname db. Vulnerable to CVE-2026-25892 (unauthenticated persistent DoS). Credentials held against 25+ combinations tested.

HighSS-013CVSS 7.5Infrastructure

PostgreSQL Directly Internet-Exposed (Port 5432)

PostgreSQL 15.5-15.6 listening on the public interface. Post-auth RCE CVEs exist (CVE-2024-10979 CVSS 8.8, CVE-2023-5869 CVSS 8.8). Credentials held against 50K+ brute-force attempts.

HighSS-014CVSS 7.5Financial

Withdrawal Transaction Injection (status=completed)

Authenticated users can INSERT withdrawal transactions with status: "completed", bypassing the intended lifecycle. No withdrawal processor currently exists — pre-staged records would activate when handler is added.

HighSS-015CVSS 8.1Authentication

2FA TOTP Secret Modifiable via API

The user_2fa table allows PATCH of totp_secret and backup_codes fields. Combined with wildcard CORS (SS-017), an attacker can remotely hijack any user's 2FA. Secrets stored in plaintext.

HighSS-016CVSS 7.5Infrastructure

Studio Dashboard Exposed (Port 3001, Zero Rate Limiting)

Supabase Studio-style dashboard accessible via HTTPS. 13,510 brute-force attempts completed in 2 minutes 5 seconds with zero rate limiting.

HighSS-017CVSS 7.5Web Security

Wildcard CORS on All Endpoints

Access-Control-Allow-Origin: * on ALL API responses including authenticated data. Any malicious website can make cross-origin requests and read responses, enabling session hijacking and data theft.

HighSS-018CVSS 6.5Gambling Integrity

All Game Probability Weights Public

Dice, roulette, limbo, plinko, and wheel probability weights, house edges, and RTPs are all readable via the anon key. Players can calculate exact expected values.

HighSS-019CVSS 7.5Access Control

settle-sports-bets Callable by Anonymous

The bet settlement Edge Function executes successfully with only the anon key. When bets exist, any anonymous user could trigger settlement.

HighSS-020CVSS 7.5Authentication

Password Change Without Old Password

Authenticated users can change their password without providing the current password. Combined with session theft (CORS/XSS), enables permanent account takeover.

HighSS-022CVSS 5.3Architecture

Zero protection layer. Origin IP 167.71.2.239 directly accessible. Vulnerable to DDoS, automated scanning, and direct service exploitation.
